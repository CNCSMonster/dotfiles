# smart-commit

智能分析 git 变更并创建合理的提交记录，使用 AI 生成提交信息。

## 描述

此命令会分析已暂存或未暂存的 git 变更，并自动：
1. 判断变更应该作为单个提交还是拆分为多个逻辑提交
2. 如果需要拆分，按功能/模块/目的对相关文件进行分组
3. 为每个组生成符合规范的提交信息
4. 执行提交操作

## 使用方法

在使用 `qodercli --with-claude-config` 时可用

在交互模式下输入 `/smart-commit` 或从命令菜单中选择。

## 选项

- `--all` 或 `-a`: 在处理前暂存所有变更
- `--dry-run` 或 `-d`: 预览提交策略但不实际提交
- `--single`: 强制单个提交（不分组）

## 工作流程

1. 分析当前 git 仓库状态
2. 检查已暂存的变更（如果使用 --all 则检查所有变更）
3. 使用 AI 确定最佳提交策略
4. 如果需要，按逻辑目的对文件分组
5. 生成符合规范的提交信息（feat、fix、chore 等）
6. 使用适当的信息创建提交

## 提示词

你是一个代码变更分析和 git 提交管理专家。你的任务是帮助用户智能地提交他们的代码变更。

**第一步：分析仓库状态**
- 检查是否在 git 仓库中
- 获取已暂存和未暂存变更的状态
- 统计新增、修改和删除的文件数量

**第二步：确定提交策略**
分析变更内容并决定：
- 应该使用单个提交还是多个提交？
- 如果是多个提交，文件应该如何分组？

考虑因素：
- 变更是否属于同一个功能或修复？
- 变更是否跨越多个不相关的模块？
- 是否存在不同的逻辑单元（例如：重构 + 新功能）？

**第三步：逻辑分组文件（如果需要拆分）**
按以下维度分组：
- 功能或特性
- 模块或目录
- 变更类型（feat vs fix vs refactor）

**第四步：生成提交信息**
为每个组创建符合规范的提交信息：
- 格式：`type(scope): subject`
- 类型：feat（新功能）、fix（修复）、docs（文档）、style（格式）、refactor（重构）、perf（性能）、test（测试）、chore（杂项）
- 主题行保持在 50 字符以内
- 如需要可添加正文说明"为什么"而不是"是什么"

**第五步：执行提交**
对每个提交组：
1. 暂存特定文件
2. 向用户展示建议的提交信息
3. 请求确认
4. 执行提交
5. 显示提交哈希和摘要

**重要准则：**
- 使用约定式提交格式（Conventional Commits）
- 编写清晰、描述性的信息
- 关注"为什么"而不是"是什么"
- 将相关变更放在一起
- 将不相关的变更分开
- 提交前向用户确认
- 为每个建议的提交显示 git diff 摘要
- 提交信息使用中文描述，但类型标签使用英文（如 feat、fix 等）

**交互示例：**

```
正在分析仓库...

已暂存的变更：
- 新增：2 个文件
- 修改：3 个文件  
- 删除：1 个文件

我已分析你的变更，建议创建 2 个提交：

提交 1：功能新增
文件：src/auth/login.js, src/auth/session.js
提交信息：feat(auth): 添加会话持久化功能

这个提交添加了会话存储功能，以便在页面重新加载后保持用户登录状态。

提交 2：代码清理
文件：src/utils/old-helper.js (删除), src/utils/helper.js (修改)
提交信息：chore: 移除废弃的辅助工具函数

移除已被新工具系统替代的旧辅助函数。

是否继续执行这些提交？(yes/no/edit)
```

**执行步骤：**
1. 首先运行 `git status --porcelain` 获取变更列表
2. 运行 `git diff --cached --stat` 查看已暂存变更的统计
3. 运行 `git diff --cached` 查看详细变更内容（如果内容太长，可以只看统计信息和文件列表）
4. 分析变更内容，确定分组策略
5. 为每组生成提交信息
6. 向用户展示计划并请求确认
7. 依次执行每个提交：
   - 如果是多个提交，使用 `git reset` 取消暂存，然后为每组依次 `git add` 和 `git commit`
   - 如果是单个提交，直接 `git commit -m "提交信息"`
8. 显示执行结果

**注意事项：**
- 如果没有已暂存的变更且没有使用 --all 选项，提示用户使用 `git add` 或重新运行时加上 `--all` 选项
- 如果用户选择 `--dry-run`，只显示计划不实际执行
- 提交信息应该简洁明了，主题行不超过 50 字符
- 如果变更较大，可以在提交信息中添加详细的正文说明
- 始终使用约定式提交格式，类型标签用英文，描述用中文
- 在执行提交前，务必向用户展示完整的提交计划并等待确认
